# ClaudeCodeGithubSetup

A template repository demonstrating two ways to integrate **Claude Code** with GitHub using a FastAPI showcase app as the example codebase.

---

## What this repo does

| Integration | Trigger | What happens |
| --- | --- | --- |
| **Interactive** | Comment `@claude` on any issue or PR | Claude reads the context and responds directly — answers questions, suggests fixes, reviews code |
| **Automated test generator** | Every pull request (opened or updated) | Claude analyzes the source code, writes pytest tests, the runner executes them, and a report is uploaded as an artifact |

```text
│   └── workflows/
│       └── code-test.yml  # Claude Code GitHub Actions workflow
├── requirements.txt
└── CLAUDE.md
```

## Local setup

```bash
# 1. Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Run the app
uvicorn app.main:app --reload
# → http://localhost:8000
# → http://localhost:8000/docs   (Swagger UI)

# 4. Run tests
pytest

# Run a single file or test
pytest tests/test_items.py
pytest tests/test_items.py::test_create_item
```

## Initial GitHub setup (one-time)

Follow these steps in order the first time you use this template.

### Step 1 — Add the Anthropic API key as a repository secret

1. Go to your repo → **Settings → Secrets and variables → Actions**
2. Click **New repository secret**
3. Name: `ANTHROPIC_API_KEY`
4. Value: your key from [console.anthropic.com](https://console.anthropic.com)

### Step 2 — Bootstrap the workflow onto `main`

The `anthropics/claude-code-action@v1` has a security validation: it checks that the workflow file on the PR branch is **byte-for-byte identical** to the version on `main`. Until the workflow file exists on `main`, it will fail with a 401 on every PR.

Merge the workflow to `main` once before opening any feature PRs:

```bash
git checkout main
git merge develop --no-ff -m "bootstrap: add Claude Code workflow"
git push origin main
```

Or merge via the GitHub UI.

### Step 3 — (Optional) Block merges on test failure

To prevent PRs from merging when tests fail:

1. Go to **Settings → Branches → Add branch protection rule**
2. Branch name pattern: `main`
3. Enable **"Require status checks to pass before merging"**
4. Search for and add: `Claude — Generate Tests / Run pytest & upload report`
5. Optionally enable **"Require branches to be up to date before merging"**

## How the GitHub Actions workflow works

### Job 1 — `Claude Interactive (@claude)`

Triggered when anyone writes `@claude` in an issue body, issue comment, PR review, or PR review comment.

Claude reads the full context and responds directly in the thread — it can answer questions, review diffs, suggest fixes, or write code.

### Job 2 — `Claude — Generate Tests / Run pytest & upload report`

Triggered on every `pull_request` event (opened or new commits pushed).

| Step | Who runs it | What it does |
| --- | --- | --- |
| Setup Python + `.venv` | Runner | Creates isolated environment, installs `requirements.txt` |
| Analyze & write tests | Claude (via `claude-code-action`) | Reads all source files, plans, writes pytest files under `tests/` |
| Run pytest | Runner (native bash) | Executes the tests, captures exit code — **this is the status check** |
| Upload artifact | Runner | Saves `test-report/pytest-report.txt` for 14 days; runs even on failure |

Splitting generation (Claude) from execution (runner) ensures:

- Pytest has a real exit code that branch protection can act on
- The report is always available as a downloadable artifact
- Claude cannot accidentally pass a broken test by re-running it

## The workflow file golden rule

> **Never include changes to `.github/workflows/code-test.yml` in a feature PR.**

If the workflow file on your PR branch differs from `main`, the OIDC security validation blocks the action with a 401. This is by design — it prevents PRs from hijacking CI permissions.

**Workflow changes must go directly to `main`** (or via a dedicated workflow-only PR that you merge without the action running):

```bash
# Make your workflow changes on a dedicated branch
git checkout -b update-workflow
# ... edit .github/workflows/code-test.yml ...
git push origin update-workflow

# Then merge directly to main (action will skip on this PR — that's expected)
# After merge, all subsequent feature PRs will use the updated workflow
```

## API overview

The FastAPI app is a patterns showcase — all storage is in-memory and resets on restart.

| Router | Prefix | Key patterns demonstrated |
| --- | --- | --- |
| `items` | `/items` | Full CRUD, pagination, search, 404 handling |
| `users` | `/users` | Dependency injection, nested routes (`/users/{id}/items`), API-key guard |
| `products` | `/products` | Response models, background tasks, category path validation |

Root routes:

- `GET /` — health check
- `GET /ping` — liveness probe
- `GET /docs` — Swagger UI (auto-generated)
